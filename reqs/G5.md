# G5 — Derived-status reminders & persistence (planned)

**Goal.** Proactively remind the team of actionable, time-driven project conditions.
Covers requirements **R22** (send-invoice), **R23** (overdue-return), **R24**
(unpaid-invoice), and **R25** (persistent fired-reminder ledger). Migrated verbatim
from the former `goals.md`; reference-doc links now point to
[../docs/eventworx API analysis.md](../docs/eventworx%20API%20analysis.md).

## Derived-status reminders (Discord, with notification)

A new class of Discord message, distinct from the existing change-notification
announcements (R15/R16). Those are **reactive and per-document**: they fire once, off a
single doc's create/status-change event, and post **silently**. Derived-status reminders
are **proactive and per-project**: they evaluate a condition computed across *all* of a
project's docs **plus the current time**, fire **with a notification (ping)**, and are
meant to prompt a human action. They are not triggered by a doc changing — most fire
precisely when *nothing* changed (a rent period elapsed, an invoice sat unpaid). The
actionable nature is why these ping and the announcements stay silent.

### Reminders to implement

Status vocabularies below are taken from the per-docType label tables in
[eventworx API analysis.md](../docs/eventworx%20API%20analysis.md) (lines 66–67) and the
aggregated sub-status fields (lines 124, 328–330). German UI label in parentheses.

- [ ] **Send-invoice reminder.** All of a project's deliverynotes are back (every `deliverynote.status` in `{returned, finished, completed}` — "Ware zurück" / "Abgeschlossen"; equivalently the order's aggregated `returnStatus == completed`) but the project has either no `finalInvoice` at all, or one still in `status == draft` (Entwurf, not yet sent). Prompts: "time to bill this job."
- [ ] **Overdue-return warning.** A project's `rentEndDate` has passed but at least one deliverynote is not yet checked in — `status` still at/before `delivered` (Geliefert) and not in `{returning, returned, finished, completed}`. EWX *also* computes a deliverynote `status == overdue` (Überfällig) and an aggregated order `returnStatus != completed` — both are ready-made signals to lean on rather than only doing our own date math. Equipment is physically still out even though downstream the system may treat it as available again. Prompts: "chase the missing return / correct the booking."
- [ ] **Unpaid-invoice reminder.** A `finalInvoice` is issued (`status` past `draft`) but not paid (`status` not `fullypaid` (Bezahlt) and aggregated `invoiceStatus != fullypaid`) for ≥ 3 weeks. EWX computes invoice `status == overdue` (Überfällig) and `reminding` (Gemahnt) itself — strongly prefer trusting those over our own threshold if they populate (see open items). If we must compute the age ourselves, the threshold is a top-of-file constant. Prompts: "follow up on payment."

### Why this needs deeper changes than the existing announcements

- **Evaluation is time-driven, not change-driven.** The incremental probe (`fetch_changed_docs`) only returns docs whose `lastModification` advanced. A rent period elapsing or an invoice aging three weeks moves no `lastModification`, so the affected project never enters the per-tick affected-set and is never re-aggregated. A derived-status pass must instead sweep **all** projects in `state.docs_by_pn` against the wall clock — either every tick, or on a slower dedicated cadence (these are day-scale conditions; a check every N minutes/hours is plenty and cheaper).
- **Conditions are computed over the full doc set of a project**, not a single `Document`. Each is a predicate over the project's deliverynotes / invoices / `rentEndDate`. Natural home: a new pure function (per project, or a field/flag on `ProjectSummary` set during `aggregate_one_project()` / `classify_project()`), kept testable and separate from the messaging.
- **Re-fire suppression is mandatory.** A naïve sweep would re-ping every tick for as long as the condition holds (unpaid invoice → a ping every minute for three weeks). Need a fired-reminder ledger keyed by `(projectNumber, reminderType)` — mirror of the `state.thread_crew_pinged` pattern. **Open question: memory-only state means a daemon restart re-fires every still-open reminder.** `thread_crew_pinged` solves the same problem by re-deriving from Discord message history on restart; decide whether to (a) re-scan the destination channel/thread for a marker, (b) accept re-fire on restart (likely too noisy for day-scale conditions that can persist for weeks), or (c) make this the first piece of genuinely persisted state. **Leaning (c)** — now elaborated as the dedicated *Persistence layer (SQLite)* below (R25), which the ledger shares with the MCP query surface.
- **Notification vs. silent.** These post **with** a ping (`flags` must *not* set `SUPPRESS_NOTIFICATIONS`, unlike the announcements), and likely an `@`-mention of a responsible role/crew so someone actually acts. Routing reuses `discord_destination()` (Technikmiete → thread, else job channel); a project with no destination is skipped as elsewhere.
- **Kill-switches & cadence config**, consistent with `ANNOUNCE_NEW_DOCS` / `ANNOUNCE_STATUS_CHANGES`: per-reminder or grouped env toggles, plus the unpaid-invoice threshold and the sweep interval as configurable constants. **DEBUG dry-run** must log `[DRY-RUN] Would remind …` like the other write paths.

### Knowledge status — mostly resolved (2026-06-22)

The status vocabularies that gate all three predicates **are now documented** in
[eventworx API analysis.md](../docs/eventworx%20API%20analysis.md) — see the per-docType
label tables (lines 66–67) and aggregated sub-status fields (lines 124, 328–330). Resolved:
- **Deliverynote return lifecycle**: `checkout`/`picking` → `delivered` → `returning` (Im Wareneingang) → `returned`/`finished` (Ware zurück) → `completed` (Abgeschlossen). "Back" = `{returned, finished, completed}`; "still out" = ≤ `delivered`. EWX also computes `overdue`.
- **Invoice lifecycle**: `draft` (not sent) → `open` (Offen, issued) → `applied` (Verrechnet) / `partiallypaid` → `fullypaid` (Bezahlt, paid); plus EWX-computed `overdue` and `reminding`.
- **Aggregated sub-statuses** are the cleanest inputs: `returnStatus` (`open`/`partial`/`completed`) for "is everything back?" and `invoiceStatus` (`open`/`prepaid`/`partiallypaid`/`fullypaid`) for "is it paid?".

**Prerequisite (code):** `_parse_doc_row` captures only the main `status`. Add
`deliveryStatus`, `returnStatus`, `invoiceStatus`, and `docSubType` to `Document` before
implementing. `docSubType` capture is already queued as **B9** (only `finalInvoice` should
count toward the invoice predicates, never a downpayment/correction).

## Persistence layer (SQLite) — queryable state + reminder ledger (R25)

The daemon is memory-only today (R4), which is fine for syncing but blocks two things:
durable re-fire suppression for the derived-status reminders above, and a future read-only
query surface for AI tooling. A single SQLite file solves both at once. **No server, no
setup** — `sqlite3` ships with Python; the "database" is one file on local disk.

**Two drivers:**
- **Reminder ledger that survives restarts** — option (c) from the reminders section. A memory-only ledger re-fires every still-open reminder on every daemon restart; for day-to-week-scale conditions that's unacceptably noisy. The fired-reminder set must persist.
- **MCP server / AI query surface (future).** We want to point OpenClaw or another AI at our project state so it can answer "status of P-1234?", "which jobs have unpaid invoices?", "what's out on rent this week?" — *without* interfacing Notion or Discord directly, which is token-inefficient (slurping pages/messages into context). A SQLite table of `ProjectSummary` rows is an indexed, cheap-to-query projection the MCP server reads instead.

**Core principle — SQLite is a *derived projection*, not new ground truth.** Eventworx
(plus the in-memory mirror rebuilt from it) stays authoritative. The DB is fully
reconstructable from a full sync, exactly like the `cache/local_*.json` snapshots — it just
happens to be read back (by the MCP server), where those never are. This preserves the
memory-only ground-truth model: losing/deleting the DB file costs nothing but a rebuild on
next full sync. (The one genuinely non-derivable table is `fired_reminders` — that's the
piece we're persisting *because* it can't be re-derived from EWX.)

**Single-writer concurrency model:**
- **The daemon is the only writer.** After each tick re-aggregates, it upserts affected rows. The MCP server (and any other reader) opens the file **read-only** (`file:state.db?mode=ro`) and only ever `SELECT`s — so SQLite's "one writer at a time" limit never comes up, and a reader can never lock out the daemon.
- **WAL mode** (`PRAGMA journal_mode=WAL`, set once — it's persistent on the file) lets readers and the writer run fully concurrently: readers never block the writer and vice versa.
- **`PRAGMA busy_timeout=5000`** on every connection so the rare lock contention retries instead of raising `SQLITE_BUSY`.
- **Caveat: same machine only.** WAL relies on real file locking / shared memory, so the DB file must live on local disk — not a network share between daemon and MCP server. Fine as long as both run on the same host.

**Proposed schema (derived rows upserted by the daemon, keyed for cheap lookup):**
- `projects` — one row per `projectNumber`: the flattened `ProjectSummary` (status, title, currentPrice, rent dates, representative job/type/URL, notionUrl, the `has_*` flags, categories as JSON/text). The MCP server's primary table.
- `documents` *(optional)* — one row per `docType:docId` for finer-grained questions ("show me P-1234's invoices"); a flattened mirror of `state.docs_by_pn`. Skip until a query actually needs doc-level detail.
- `fired_reminders` — `(project_number, reminder_type)` PK, plus `fired_at`. The durable ledger; UPSERT on fire, read on the reminder sweep to suppress re-fires. **The only non-derived table.**
- `sync_meta` *(optional)* — last full-sync / checkpoint timestamps, mostly for the MCP server to report freshness ("data as of …").

**Implementation checklist:**
- [ ] Add a `state.db` path constant (top-of-file, alongside `SYNC_STATE_FILE`); git-ignore it like `cache/`.
- [ ] Open the writer connection once at startup; set `journal_mode=WAL` + `busy_timeout`. Create tables if absent (idempotent `CREATE TABLE IF NOT EXISTS`).
- [ ] Upsert `projects` (and optionally `documents`) rows for affected PNs inside `push_projects()` / after aggregation — same affected-set the Notion/Discord push already walks.
- [ ] Load `fired_reminders` into memory at startup (replaces the restart re-derivation question); write through to the DB whenever a reminder fires. The in-memory dict stays the fast path; the DB is the durable backing.
- [ ] **DEBUG/dry-run:** decide whether DB writes happen under `DEBUG=true` or are gated like the external writes. Leaning *let them happen* — it's local, not an external side effect, and a populated dry-run DB is useful for inspecting/testing the MCP queries. Document whichever we pick.
- [ ] Update the sync-model docs (`info.md`) and `CLAUDE.md` once landed: the daemon is no longer strictly memory-only — it maintains a derived, rebuildable SQLite projection (and one genuinely persisted `fired_reminders` table).

**Open questions:**
- [ ] **One file or two?** A single `state.db` for both the derived projection and the ledger is simplest. Only reason to split: if we ever want to wipe/rebuild the projection without touching the ledger (a `DROP TABLE projects` achieves the same within one file, so probably not worth two files).
- [ ] **Does the MCP server need its own schema/view layer**, or does it query the daemon's tables directly? Direct is fine to start; add read-only `VIEW`s if the AI-facing shape diverges from the storage shape.
- [ ] **MCP server lives in this repo or a sibling?** It shares only the DB file (read-only), so it can be a fully separate process/repo — no import coupling to `ewxSync.py`. Decide when we actually build it.

## Acceptance criteria (per requirement)
All must hold before the matching R is marked done in `reqs.md`.

- [ ] **R22** — Send-invoice reminder fires exactly once per project when all deliverynotes are back and no sent final invoice exists; suppressed thereafter.
- [ ] **R23** — Overdue-return warning fires when `rentEndDate` has passed and equipment is not checked in; suppressed once resolved.
- [ ] **R24** — Unpaid-invoice reminder fires when a final invoice is unpaid for ≥ the configured threshold; suppressed once paid.
- [ ] **R25** — A daemon restart does not re-fire any reminder that was already fired and is still open (ledger persisted to SQLite).
- [ ] All reminders ping (no `SUPPRESS_NOTIFICATIONS`), respect per-reminder kill-switches, and log `[DRY-RUN] Would remind …` under `DEBUG`.
